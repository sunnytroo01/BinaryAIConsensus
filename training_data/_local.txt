 .local is a special-use top-level domain (TLD) reserved exclusively for Multicast DNS (mDNS), a networking protocol that facilitates the resolution of hostnames to IP addresses within small, local networks lacking a conventional unicast DNS server. Defined in RFC 6762, published in February 2013 by the Internet Engineering Task Force (IETF), .local enables zero-configuration networking by allowing devices to use simple names like "hostname.local" that are meaningful only on the originating local link. This domain is not delegated or registrable in the global Domain Name System (DNS) and operates under special semantics to prevent conflicts with public internet domains.
In mDNS, queries for names ending in ".local." are multicast to the link-local addresses 224.0.0.251 (IPv4) or FF02::FB (IPv6) on UDP port 5353, allowing all devices on the network to respond if they match the query. Hostnames under .local must be unique within the local link; devices perform name probing upon startupsending multicast queries with random delays of 0250 milliseconds, repeated up to three times if neededto detect and resolve conflicts by deferring to existing names or selecting alternatives. Resource records for .local names, such as A (IPv4 address) or AAAA (IPv6 address), typically have a recommended time-to-live (TTL) of 120 seconds to ensure timely updates in dynamic environments.
The protocol supports both IPv4 and IPv6, with multihomed devices defending their .local names across all interfaces while permitting duplicates on separate links. Developed initially for services like Apple's Bonjour and open-source implementations such as Avahi, mDNS with .local has become a standard for service discovery in home and small office networks, including printer sharing, media streaming, and IoT device connectivity. IANA maintains .local in its registry of special-use domain names, prohibiting its use outside mDNS contexts to avoid interference with global DNS operations.

Overview

Definition and Reservation

The .local domain is a special-use domain name reserved by the Internet Engineering Task Force (IETF) for local networking purposes, explicitly prohibited from being installed as a top-level domain in the global Domain Name System (DNS) root zone to avoid conflicts with public DNS operations. This reservation ensures that .local names are resolved solely within local networks without requiring a global DNS infrastructure.
Complementing the general framework for special-use domains in RFC 6761, published in 2013, which defines a registry of such domains intended for specific, non-global uses, RFC 6762 from the same year specifically designates .local for link-local multicast name resolution, enabling devices on the same network segment to discover and resolve names without a central DNS server. This builds on earlier IETF reservations of other domains like .test, .example, and .invalid in RFC 2606 (1999) for documentation, testing, and illustrative examples in technical specifications, predating .local's dedicated role in multicast DNS (mDNS).
The Internet Assigned Numbers Authority (IANA) maintains .local in its Special-Use Domain Names registry, a definitive list that reinforces its non-delegable status and guides implementers to treat it as reserved for private, link-local applications rather than public registration. This registry entry, which references RFC 6762, underscores the IETF's commitment to preserving .local for zeroconf networking scenarios, such as mDNS-based service discovery on local links.

Purpose in Local Networking

The .local domain serves as a special-use namespace reserved for zero-configuration networking (zeroconf) on local links, allowing devices to automatically resolve hostnames such as "hostname.local" through multicast queries without relying on a central DNS infrastructure. This designation enables seamless name resolution in environments where traditional unicast DNS servers are absent or impractical, designating a portion of the DNS namespace for local use free from global delegation requirements or fees. Unlike conventional DNS names, those ending in ".local." possess only local significance, ensuring they do not conflict with the global Domain Name System.
Primarily employed in peer-to-peer scenarios within home, office, or small-scale networks, .local facilitates the discovery of services like printers, media servers, and file shares by allowing devices to advertise and query for each other dynamically. This approach supports effortless connectivity among compatible devices, such as smartphones, laptops, and peripherals, by leveraging multicast DNS (mDNS) for distributed resolution on the same local link. For instance, a user can connect to a network printer simply by addressing it as "printer.local," bypassing the need for static IP assignments or manual service registration.
The domain accommodates both IPv4 and IPv6 address resolution protocols, providing a unified mechanism for name-to-address mapping without manual IP configuration or a dedicated DNS server. This dual-stack compatibility ensures broad applicability across modern networks, where devices may transition between address families seamlessly.
A primary benefit of .local lies in simplifying connectivity within dynamic environments, such as Wi-Fi networks where devices frequently join or leave the link, minimizing administrative overhead and maintaining functionality even during infrastructure disruptions. By enabling self-configuring name resolution, it reduces setup complexity for non-expert users in transient or ad-hoc setups, promoting plug-and-play interoperability.

Historical Development

Early IETF Reservations

In June 1999, the IETF published RFC 2606, which reserved several top-level domains including .example and .invalidalongside .test and .localhostto prevent conflicts arising from their use in documentation, software examples, and testing environments. This Best Current Practice (BCP 32) aimed to designate these names as unavailable for registration in the global DNS, thereby avoiding situations where local or illustrative uses could inadvertently collide with production deployments.
Concurrently, in the same month, IETF working group draft-ietf-dnsind-local-names-07 proposed the reservation of .local as a new top-level domain specifically for local private DNS zones, mirroring the approach of reserved private IP address blocks in RFC 1918 and RFC 2373. The primary purpose was to provide a safe namespace for internally administered networks, preventing accidental global registrations that might lead to resolution failures, namespace collisions, or security vulnerabilities in distributed systems. Although this draft expired without advancement to RFC status in December 1999, .local was not formally reserved at that time and subsequently became commonly used in private networks, such as for internal Active Directory domains as recommended by Microsoft until around 2013. This widespread private use highlighted the need for clarification and contributed to later conflicts with emerging local networking protocols.
These early reservations established a foundational framework for non-global domain usage, influencing subsequent developments such as the integration of .local into Multicast DNS (mDNS) standards for zero-configuration networking.

Standardization of mDNS

The formal standardization of Multicast DNS (mDNS) occurred through RFC 6762, published in February 2013 by the Internet Engineering Task Force (IETF) on the Standards Track. Authored by Stuart Cheshire and Marc Krochmal of Apple Inc., this document defines mDNS as a protocol enabling DNS-like operations on the local link without relying on a conventional unicast DNS server infrastructure. It explicitly mandates the use of the .local top-level domain for all link-local hostname resolutions, designating names ending in ".local." as having exclusively local significance to facilitate zero-configuration networking, and reserves .local specifically for mDNS use in the global DNS.
This standardization built upon foundational work from the IETF Zeroconf Working Group, chartered in September 1999 to address zero-configuration networking challenges. Throughout the 2000s, the group produced numerous drafts exploring related concepts, including early proposals for multicast-based name resolution that informed mDNS development; for instance, individual submissions like draft-cheshire-dnsext-multicastdns evolved over iterations from 2002 onward, culminating in the formal RFC. These efforts addressed the need for automatic host naming and discovery in small networks without manual administration, laying the groundwork for mDNS as a key component of Zeroconf protocols.
RFC 6762 specifies that queries for .local names must utilize link-local multicast addressingIPv4 address 224.0.0.251 or IPv6 address FF02::FB on UDP port 5353rather than unicast queries to global DNS servers, ensuring resolutions remain confined to the local link. This requirement prevents unintended leakage of local queries to external resolvers and promotes efficient, peer-to-peer discovery among devices.
By codifying these mechanisms, RFC 6762 provided a unified specification for mDNS, addressing prior informal implementations from Zeroconf drafts and proprietary systems to ensure greater interoperability across diverse devices and operating systems. This standardized mDNS for consistent deployment in local networks. It also integrates briefly with DNS Service Discovery (DNS-SD, defined in RFC 6763) to enable service enumeration using .local names via multicast queries.

Technical Specifications

Multicast DNS Protocol

Multicast DNS (mDNS) enables IP address resolution for hostnames within a local network link without relying on a unicast DNS infrastructure. The protocol specifically utilizes the .local top-level domain suffix for names that are significant only on the local link, allowing devices to discover each other's IP addresses through multicast communication. mDNS messages are transported over UDP using port 5353 for both queries and responses. Queries and responses are sent to the multicast address 224.0.0.251 in IPv4 networks and ff02::fb in IPv6 networks, ensuring they remain confined to the local subnet.
In operation, a device seeking to resolve a .local hostname multicasts a query to the appropriate address on port 5353, specifying the desired name and query type in the DNS message format. Any device on the same link that recognizes the name responds via multicast (or unicast if requested) with the corresponding resource records, primarily A records for IPv4 addresses or AAAA records for IPv6 addresses. This peer-to-peer response mechanism allows immediate resolution without a central authority, with responses carrying a TTL of 120 seconds for cached records to balance freshness and network efficiency.
To maintain name uniqueness, mDNS incorporates conflict detection through a probing process. When a device intends to claim a hostname, it first sends up to three probe queries with a random delay (250 milliseconds apart) to check for existing usage of that name. If another device detects the probe and has a conflicting record, both perform tie-breaking, with the device having the numerically smaller IP address winning; if IP addresses are the same, the earlier probe transmission wins, and if still tied, the numerically smaller MAC address wins. The loser defers and may select an alternative name, such as appending a numeric suffix. This proactive mechanism prevents duplicate assignments and ensures stable local resolution.
The protocol supports standard DNS resource record types relevant to local discovery, including A and AAAA for address mapping, PTR for reverse lookups, and SRV for service location, while adapting them to the multicast environment. mDNS messages are structured to avoid IP fragmentation by fitting within the interface's MTU (typically 1500 bytes for Ethernet, minus headers), with a maximum recommended size of 9000 bytes for jumbo frames but emphasizing compact packetsideally under 576 bytes for broad compatibilityto minimize overhead and ensure reliable delivery on diverse local networks. mDNS is often used alongside DNS Service Discovery (DNS-SD) for broader service enumeration on .local domains.

DNS Service Discovery Integration

DNS-Based Service Discovery (DNS-SD), as specified in RFC 6763 published in 2013, extends the capabilities of Multicast DNS (mDNS) by enabling the advertisement and discovery of network services on local links using the .local domain. This integration allows devices to locate and connect to services such as web servers via service types like _http._tcp.local without requiring manual configuration or a centralized directory. By leveraging mDNS as the underlying transport mechanism, DNS-SD operates over multicast UDP on port 5353, facilitating zero-configuration networking in environments like home or office LANs.
Services are registered by publishing specific DNS resource records under the .local domain, primarily using SRV records to specify the target hostname and port number for the service, and TXT records to provide additional metadata such as version information or configuration parameters. For instance, a printing service might register an instance named "OfficePrinter._ipp._tcp.local", where the SRV record points to the printer's hostname and port (e.g., 631 for IPP), while the TXT record could include details like supported formats or capabilities. PTR records link service types to their instances, allowing clients to enumerate available options for a given protocol. This structure ensures that service discovery remains compatible with standard DNS semantics while adapting to multicast environments.
Browsing for services begins with a client issuing a multicast PTR query to _services._dns-sd._udp.local, which returns a list of available service types on the local network, such as _ipp._tcp or _ssh._tcp. Once a desired service type is identified, the client queries for PTR records of that type (e.g., _ipp._tcp.local) to retrieve instances, followed by SRV and TXT queries for each instance to obtain connection details. This query process uses wide-area queries in mDNS to cover the local link efficiently.
DNS-SD's design enables dynamic service registration and deregistration without a central authority, as devices can announce their services directly via mDNS announcements and handle conflicts through probing and renaming mechanisms. This complements basic mDNS hostname resolution by focusing on service-oriented discovery, promoting seamless interoperability in ad-hoc networks.

Platform Implementations

Apple Bonjour

Bonjour, introduced in 2002 as part of Mac OS X 10.2 under the name Rendezvous, provides native support for Multicast DNS (mDNS) resolution of .local hostnames on Apple devices. This implementation enables zero-configuration networking, allowing devices to automatically discover and connect to each other on local networks without manual IP address or DNS setup. Bonjour adheres to IETF standards for mDNS, facilitating seamless service advertisement and query responses using the .local top-level domain.
Integrated natively into iOS, macOS, and tvOS, Bonjour powers key ecosystem features that rely on .local hostnames for local discovery. For instance, it enables AirPlay for streaming media to compatible devices like Apple TV and AirPort base stations by advertising and resolving services via mDNS. Similarly, AirDrop uses Bonjour for initial peer discovery in file sharing between nearby Apple devices, combining it with Bluetooth Low Energy and Apple Wireless Direct Link for secure transfers. Printer discovery through AirPrint also depends on Bonjour to locate and connect to compatible printers on the local network without additional drivers.
A notable feature of Bonjour is the Sleep Proxy mechanism, which allows dormant devices to maintain network discoverability while conserving power. When a device enters sleep mode, an awake proxysuch as another Mac or Apple TVregisters and responds to mDNS queries on its behalf using stored service records, including proxying Address Resolution Protocol (ARP) requests to wake the device only when necessary. This supports Wake on Demand for services like file sharing and printing, ensuring uninterrupted availability in low-power states.
While core components like the mDNSResponder are open-source under the Apache 2.0 license, Bonjour includes proprietary extensions optimized for Apple's ecosystem integration. The open-source elements, available via Apple's Darwin project, allow developers to build compatible services, but full seamless operation with Apple-specific features like Handoff and Continuity relies on the closed-source aspects.

Linux Avahi

Avahi is the primary open-source implementation of Multicast DNS (mDNS) and DNS Service Discovery (DNS-SD) for Linux and other Unix-like systems, enabling seamless use of the .local domain for local network hostname resolution and service discovery. Released on August 22, 2005, as version 0.1, it was developed to provide a free alternative to proprietary solutions, supporting both IPv4 and IPv6 from its inception and integrating directly with the mDNS protocol for zero-configuration networking. It is included by default in major Linux distributions such as Ubuntu and Fedora, facilitating cross-platform compatibility with devices using Apple's Bonjour.
Configuration of Avahi is managed through the primary file /etc/avahi/avahi-daemon.conf, which allows customization of settings like domain suffixes, protocol preferences, and publishing behaviors, while service definitions are placed in XML files under /etc/avahi/services. For hostname resolution in the .local domain, Avahi integrates with the Name Service Switch (NSS) via the libnss-mdns module, which must be enabled in /etc/nsswitch.conf by adding "mdns4_minimal " or similar to the hosts line, allowing standard tools like ping to resolve local hostnames without manual IP configuration.
Avahi supports publishing arbitrary services on the local network, such as Samba file shares advertised as _smb._tcp (e.g., via a .service file defining the service type, port, and TXT records for share details), enabling automatic discovery by clients like macOS Finder or Windows Network Browser. It also provides browsing capabilities for discovering published services across platforms, with full IPv6 support ensuring dual-stack environments function reliably for both resolution and advertisement. Command-line tools like avahi-browse allow manual querying of available services, for example, running avahi-browse -a to list all local mDNS announcements, aiding in debugging and exploration without graphical interfaces.

Microsoft Windows Support

Microsoft Windows provides partial native support for Multicast DNS (mDNS) and the .local top-level domain starting with Windows 10 version 1703 (Creators Update) released in April 2017, enabling basic hostname resolution in certain scenarios such as Miracast device discovery. However, this implementation is limited and does not fully support DNS Service Discovery (DNS-SD) or comprehensive .local name resolution without additional components. For complete functionality, including printer discovery and service browsing on .local networks, third-party tools like Apple's Bonjour Print Services for Windows are required, which installs the necessary mDNS responder and service discovery libraries.
Windows 11 builds upon this partial support, incorporating mDNS for peer-to-peer discovery in features like Azure Sphere applications and wireless display protocols, but .local resolution remains inconsistent, often failing for complex service advertisements without add-ons. Windows Server editions, including Server 2019 and 2022, lack native mDNS support entirely, relying on third-party implementations for interoperability with .local-based devices. In practice, full .local resolution across Windows client and server environments typically necessitates installing Bonjour or equivalent software to bridge gaps in native capabilities.
In enterprise settings, mDNS support is frequently disabled to prevent conflicts with Active Directory DNS, particularly when domains use the .local suffix, as mDNS queries can interfere with authoritative DNS resolution and introduce security risks like name poisoning. Microsoft recommends controlling mDNS traffic via firewall rules or registry modifications in such environments to maintain domain integrity.
As of 2025, Windows 10 and 11 continue to recommend third-party add-ons like Bonjour Print Services for robust .local interoperability with Apple and Linux devices, despite incremental native improvements in client editions. This approach ensures compatibility in mixed ecosystems while mitigating the limitations of built-in mDNS functionality.

Vendor Recommendations and Configurations

Microsoft Guidelines

In the early 2000s, Microsoft documentation and practices commonly accommodated the use of .local as a top-level domain (TLD) for private Active Directory (AD) domains, particularly in isolated internal networks where internet routability was not required. This approach was reflected in knowledge base articles and deployment guides from that era, such as those addressing domain naming conventions for small business servers. However, as hybrid cloud integrations became prevalent, Microsoft shifted its stance.
By 2017, with the widespread adoption of Azure AD Connect for synchronizing on-premises AD with Microsoft Entra ID (formerly Azure AD), official guidance began emphasizing the risks of .local domains. Documentation from this period, including Microsoft Learn resources updated through 2021, explicitly advised against using .local for new AD deployments due to potential resolution conflicts, particularly with Multicast DNS (mDNS) protocols used by devices for local service discovery. These conflicts can arise when .local hostnames overlap between AD DNS and mDNS queries on the same network.
As of 2025, Microsoft recommends using routable, registered public domains (e.g., subdomains like corp.example.com) for AD forests in hybrid environments to ensure seamless integration with cloud services and avoid synchronization issues. Alternatives such as .internal are suggested in some contexts for strictly internal use, but only if they do not conflict with established protocols; the priority remains on verifiable, owned domains to support features like certificate issuance and federation.
A key incompatibility highlighted in current guidelines is the use of .local as a User Principal Name (UPN) suffix in AD, which prevents proper directory synchronization with Microsoft 365. During sync via Azure AD Connect, UPNs ending in .local (e.g., ) are automatically rewritten to the tenant's .onmicrosoft.com suffix (e.g., ), leading to mismatches with verified domains and blocking features like single sign-on and email routing. To resolve this, administrators must add matching UPN suffixes in AD Domains and Trusts and update user attributes accordingly, often via PowerShell scripts.

Linux and Other Configurations

On Linux systems, enabling Multicast DNS (mDNS) resolution for .local domains requires installing the nss-mdns package, which provides the necessary Name Service Switch (NSS) module for host resolution. To prioritize local mDNS resolution, edit the /etc/nsswitch.conf file and append "mdns4_minimal " to the hosts line, ensuring that .local queries are handled by mDNS before falling back to other resolvers. This configuration allows commands like ping hostname.local to resolve addresses via mDNS without external DNS interference.
The Avahi daemon, which implements mDNS on Linux, must be started and enabled using systemd with the command systemctl enable --now avahi-daemon.service. In the configuration file /etc/avahi/avahi-daemon.conf, set domain=.local under the section to specify the default domain for service announcements, and configure publish-domain=false under the section to prevent the daemon from announcing the .local domain to other networks, thereby avoiding unintended global exposure of local services. After modifications, restart the daemon with systemctl restart avahi-daemon to apply changes.
For non-Microsoft platforms like Android, mDNS support is provided through the mDNSResponder component integrated into the DNS Resolver module, enabling .local resolution without additional user configuration on devices running Android 12 or later. On embedded devices such as IoT routers or single-board computers, ensure that firewalls permit UDP port 5353 for multicast traffic (224.0.0.251 for IPv4 and ff02::fb for IPv6) to allow mDNS packets within the local network.
A recommended best practice for securing .local usage across Linux and other platforms is to isolate mDNS traffic to the local subnet using access control lists (ACLs) on routers or switches, blocking outbound multicast packets to prevent external queries or leaks beyond the intended network segment. This can be achieved by configuring rules to drop traffic destined for 224.0.0.251 on inter-subnet interfaces, ensuring mDNS remains confined to trusted local environments.

Issues and Alternatives

Global DNS Conflicts

When .local domains are used for internal unicast DNS in enterprise environments, queries intended for local resolution can inadvertently leak to external or global DNS resolvers if network configurations do not properly isolate them, resulting in NXDOMAIN (non-existent domain) responses from public servers that do not recognize .local as a valid top-level domain. This leakage occurs because .local is reserved exclusively for Multicast DNS (mDNS) under RFC 6762, and standard DNS resolvers treat it as non-authoritative, often triggering unintended fallbacks to mDNS protocols that may fail to resolve unicast records or expose local network details. In hybrid networks combining on-premises systems with cloud services, such misconfigurations exacerbate resolution failures, as devices roaming outside the local network attempt to resolve .local names via upstream resolvers, leading to timeouts or erroneous mDNS multicasts that do not propagate globally.
Recent incidents in 2024 have underscored these risks in hybrid setups, where internal .local usage collides with global DNS expectations, causing widespread resolution disruptions; for instance, a Krebs on Security report detailed namespace collisions in enterprise networks, highlighting how non-routable internal domains like .local contribute to credential exposure and service outages when queries escape local boundaries. In one documented case involving a similar namespace collision, over 140,000 unintended queries were observed in a single week across affected systems. These events, often tied to legacy Active Directory deployments, illustrate how .local's non-routable nature amplifies conflicts in mixed on-premises and remote access environments. In 2025, Windows Server 2025 deployments have introduced additional challenges, including domain controller failures to apply domain firewall profiles after restarts and trust relationship errors on client devices, further complicating .local-based Active Directory environments.
During cloud migrations, .local-based Active Directory domains frequently fail to synchronize with services like Microsoft Entra ID (formerly Azure AD) due to their non-routable status, which prevents proper user principal name (UPN) matching and authentication federation. Microsoft documentation notes that .local domains cannot be directly verified in Entra ID, leading to sync errors where objects are not provisioned or attributes like email addresses mismatch, requiring manual UPN suffix additions in AD DS to align with verified public domains. This issue has been prevalent in hybrid identity setups, where attempting to sync .local users results in duplicate entries or failed joins, complicating migrations to cloud-only environments and necessitating domain renames or alternate UPNs to restore functionality.
To mitigate these global DNS conflicts, organizations can implement split-DNS architectures, which direct .local queries exclusively to internal resolvers while forwarding external domains to global servers, preventing leaks and ensuring authoritative local responses. Conditional forwarders in DNS servers like Windows Server can be configured to block .local traffic from reaching upstream resolvers, routing it instead to on-premises domain controllers or mDNS-enabled services, thereby avoiding NXDOMAIN responses and unintended fallbacks. Microsoft guidelines emphasize this approach for hybrid networks, recommending split-DNS alongside verified domain UPNs to maintain resolution integrity during cloud integrations.

Security and Privacy Concerns

Multicast DNS (mDNS), which powers the .local domain for local network service discovery, inherently broadcasts queries and responses across the local network using UDP port 5353 and multicast addresses such as 224.0.0.251 for IPv4 or FF02::FB for IPv6, allowing any device on the same link to intercept this traffic. This exposure can reveal device names, service types, and other metadata to unauthorized listeners, such as those using packet capture tools like Wireshark, potentially aiding reconnaissance by attackers on the network.
A primary security vulnerability stems from the absence of built-in authentication or encryption in mDNS, enabling spoofing attacks where a malicious device can forge responses to .local queries, directing traffic to false IP addresses and facilitating man-in-the-middle (MitM) interceptions or service hijacking. Such attacks exploit the protocol's trust in local link participants, with no inherent verification of response origins beyond basic source address checks that can be bypassed on shared networks.
From a privacy perspective, the automatic advertisement of services via mDNS announcements discloses network topology, including active devices and available resources, which is particularly risky on untrusted or open Wi-Fi networks where eavesdroppers can map the environment without authentication. This passive leakage can contribute to fingerprinting or targeted attacks, as the protocol does not encrypt payloads or restrict visibility within the local scope.
As of November 2025, mDNS has been implicated in amplification DDoS attacks when misexposed to the internet, and specific vulnerabilities in implementations have been disclosed, such as buffer overflows (e.g., CVE-2025-5688 in FreeRTOS-Plus-TCP) and crashes via malformed packets (e.g., CVE-2025-24252 in AirPlayReceiver). Multicast DNS spoofing remains a top common cybersecurity vulnerability. However, no widespread exploits targeting standard .local usage on isolated local networks have been reported in major security databases. To mitigate these risks, experts recommend network segmentation via VLANs or firewalls to confine mDNS traffic, disabling unnecessary service publications on devices, and applying host-based controls like Windows Defender Firewall rules to limit exposure.

Alternative Private Domains

To address conflicts with the global Domain Name System (DNS) that can occur when using .local for private networking, the Internet Engineering Task Force (IETF) and Internet Corporation for Assigned Names and Numbers (ICANN) have designated specific top-level domains (TLDs) for non-public use.
The .home.arpa domain, defined in RFC 8375 and published in May 2018, is reserved for non-unique use in residential home networks, enabling multiple independent home networks to employ the same domain without global DNS collisions. This special-use domain supports resolution via both unicast DNS for traditional name queries and multicast DNS (mDNS) through integration with the Home Networking Control Protocol (HNCP), which facilitates automated configuration and service discovery in home environments.
In August 2024, ICANN formally reserved the .internal TLD for private-use applications following a board resolution on July 29, 2024, ensuring it is permanently excluded from delegation in the DNS root zone. This makes .internal particularly suitable for enterprise internal networks, as it operates exclusively within unicast DNS setups without dependency on mDNS, reducing risks of interference with public internet resolution while providing a standardized namespace for corporate intranets.
Other non-standard options, such as .lan or .corp, have been informally adopted for private networks but lack formal IETF or ICANN reservations, potentially leading to future delegation risks in the public DNS. Microsoft recommends avoiding single-label domain names (e.g., simply "lan" without a suffix) due to limited support, compatibility issues with applications, and challenges in Active Directory integration.
Transitioning from .local to an alternative like .home.arpa or .internal involves establishing a new Active Directory forest with the chosen domain, using the Active Directory Migration Tool (ADMT) to transfer users, groups, computers, and SID history while preserving access permissions. Best practices as of 2025 include updating User Principal Name (UPN) suffixes in Active Directory to match the new domain, reconfiguring DNS resolvers and forwarders on domain controllers and clients to prioritize the internal zone, and testing hybrid connectivity before decommissioning the old .local infrastructure to minimize downtime.
