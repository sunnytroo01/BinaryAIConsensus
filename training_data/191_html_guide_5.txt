HTML Best Practices and Common Mistakes.

Always use the DOCTYPE declaration. Without it, browsers enter quirks mode and render pages inconsistently. <!DOCTYPE html> is all you need for HTML5.

Always set the character encoding. <meta charset="UTF-8"> should be the first element inside head. Without it, special characters may display incorrectly.

Always include the viewport meta tag for mobile devices. <meta name="viewport" content="width=device-width, initial-scale=1.0"> Without this, mobile browsers will render the page at desktop width and zoom out, making text tiny and unreadable.

Use semantic elements instead of div for everything. Do not write <div class="header">. Write <header>. Do not write <div class="nav">. Write <nav>. Do not write <div class="footer">. Write <footer>. Semantic elements improve accessibility, SEO, and code readability.

Never skip heading levels. Do not go from h1 to h3 without an h2 in between. Screen readers and search engines use heading hierarchy to understand page structure. Use only one h1 per page.

Always include alt text on images. Empty alt="" is acceptable for decorative images but every informational image needs descriptive alt text. Do not put "image of" or "photo of" in alt text since screen readers already announce it as an image.

Do not use br tags for spacing. Use CSS margin and padding instead. The br tag is for line breaks within text, like addresses or poetry, not for creating vertical space between elements.

Do not use tables for layout. Tables are for tabular data only. Use CSS Flexbox or Grid for page layout. Using tables for layout breaks accessibility because screen readers interpret them as data tables.

Do not use inline styles unless absolutely necessary. Keep styles in a separate CSS file or in a style tag in the head. Inline styles are hard to maintain, cannot be cached by the browser, and make the HTML harder to read.

Close all tags properly. HTML5 is forgiving and will often render correctly with unclosed tags, but unclosed tags can cause subtle layout bugs and break accessibility tools. Write clean, valid HTML.

Use lowercase for all tag names and attributes. HTML is case-insensitive but convention is lowercase. <div class="container"> not <DIV CLASS="container">.

Quote all attribute values. <input type="text" class="form-input"> not <input type=text class=form-input>. Unquoted attributes can cause parsing issues with certain values.

Validate your HTML. Use the W3C Markup Validation Service at validator.w3.org to check for errors. Valid HTML renders more consistently across browsers and is more accessible.

Keep your HTML clean and indented consistently. Use two or four spaces for indentation. Nested elements should be indented one level deeper than their parent. Clean HTML is easier to read, debug, and maintain.

Do not put block elements inside inline elements. A div inside a span or a p inside an a tag is invalid and can cause rendering issues. Block elements go inside other block elements. Inline elements go inside block elements.

Use the button element for clickable actions, not div or span with click handlers. Buttons are keyboard-accessible by default, can be focused with tab, and activated with Enter or Space. Custom clickable divs require additional ARIA attributes and JavaScript to achieve the same accessibility.

Use the correct input types for the data you are collecting. type="email" provides email validation and shows the email keyboard on mobile. type="tel" shows the phone keyboard. type="number" shows numeric input with increment buttons. type="date" shows a date picker. Using the correct type improves user experience on all devices.
